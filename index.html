Abstract

WordleBot — autonomous robotic Wordle

WordleBot RS2 is a fully autonomous robotic system that plays the word game Wordle. An Intel RealSense D435i depth camera, mounted on a UR3e collaborative robot arm, scans a workspace of lettered blocks. A custom CNN classifies each letter in real time, and TF2 transforms project their positions from camera frame into the robot's world frame. A behaviour-tree orchestrates the game loop: the word-solver subsystem selects an optimal 5-letter guess, the high-level controller plans a pick-and-place trajectory for each letter, and the robot spells the word on the board. Wordle feedback is received, the candidate word list is pruned, and the loop repeats — all without human intervention.

The project is implemented entirely in ROS 2 (Humble) with Python, spanning four tightly-integrated subsystems: perception & mapping (SS1), pick-and-place control (SS2), behaviour-tree orchestration (SS3), and game logic & gamification (SS4).

System design

ROS 2 subsystem architecture

All four subsystems communicate via ROS 2 topics and services. SS1 publishes detections every frame while scanning; the completed gameboard is latched once and consumed by SS2. The behaviour tree (SS3) coordinates transitions, ensuring the robot never moves while the camera is reading.

SS1 · Luke Klusman

Perception & Mapping

  • RealSense D435i (colour + depth)
  • White-card / BLOCK detection modes
  • CNN: 36-class LetterCNN (A–Z, 0–9)
  • Temporal majority-vote smoothing (15 frames)
  • Dot-based block orientation (±180° disambiguation)
  • TF2 camera→gripper_tcp→world projection
  • Publishes: /perception/detections, /perception/gameboard_state
SS2 · Connor

High-Level Control & Pick & Place

  • Subscribes to /perception/gameboard_state
  • Subscribes to /hl_control/word_request
  • Plans UR3e trajectories for each block
  • OnRobot RG2 gripper control
  • Deposits letters at target board positions
SS3 · Elijah

Behaviour Tree & Safety

  • Publishes /mission/state (SCANNING / IDLE)
  • Subscribes to /perception/human_detected
  • Safety stop on human intrusion
  • Coordinates scan → solve → place game loop
  • Scene-change triggered re-scan logic
SS4 · Kermit

Game Logic & Gamification

  • 5-letter dictionary with candidate filtering
  • Entropy-based best-guess selection
  • Mode A (AI plays) / Mode B (human plays)
  • Letter-hunt confirmation: picks word first, then verifies all blocks are visible
  • Publishes: /hl_control/word_request, /diagnostics

Key ROS 2 topics

Topic Type Publisher Description
/perception/detections String (JSON) SS1 Per-frame letter detections with camera-frame 3D positions
/perception/gameboard_state GameboardState (latched) SS1 World-frame poses for all detected blocks, published once per scan
/mission/state String SS3 SCANNING / IDLE — controls perception scan window
/hl_control/word_request String (latched) SS4 5-letter guess word; triggers pick-and-place planning in SS2
/perception/human_detected Bool SS1 Safety signal to SS3 — robot stops if a human enters the workspace

Results

CNN evaluation & system performance

The LetterCNN was trained on 36 classes (A–Z, 0–9) collected using the custom collect_training_data.py pipeline. The preprocessing chain — grayscale conversion, 64×64 resize, histogram equalisation — is matched exactly between training and inference to avoid distribution shift.

Discussion

Reflection & future work

The perception pipeline achieved reliable letter recognition under varied lighting by combining a depth-gated contour detector with temporal majority-vote smoothing over 15 frames. The dot-based orientation estimator disambiguated the 180° symmetry that cv2.minAreaRect alone cannot resolve, which proved critical for letters like N, Z, and d/q that are orientation-sensitive.

Hand-eye calibration remains the single most important step before demo day — manually estimated mount offsets compound into pick-and-place failures on 5 cm blocks. The Wednesday calibration session using ArUco ID 365 and the handeye_realsense package will replace the six guessed CAM_MOUNT_* values with ground-truth measurements.

⚠ Calibration reminder: Print the ArUco marker (ID 365, 150 mm physical size) before the Wednesday lab session. Both realsense_info.yaml and realsense_camera_cnn.py (lines 110–113) must be updated with real intrinsics from the same camera at 640×480.

Future work includes: extending the CNN to handle partially-occluded letters using attention mechanisms; implementing a hard-coded "ARISE" opening guess known to maximise information gain; and adding multi-block duplicate handling so the robot can place two copies of the same letter (e.g., APPLE requires two P blocks).

Reproducibility

Install & run

Tested on Ubuntu 22.04, ROS 2 Humble, Python 3.10.

# Clone and build
git clone https://github.com/YOUR_ORG/YOUR_REPO.git ~/wordlebot_ws/src/wordlebot
cd ~/wordlebot_ws
colcon build --packages-select hl_control wordlebot_perception

# Install Python dependencies
pip install torch torchvision opencv-python pyrealsense2 mediapipe

# Launch RealSense driver (Terminal 1)
ros2 launch realsense2_camera rs_launch.py enable_depth:=true align_depth.enable:=true

# Launch perception node (Terminal 2)
python3 realsense_camera_cnn.py

# Controls: SPACEBAR = toggle scanning · Q = quit
